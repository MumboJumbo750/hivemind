# Glossar — Hivemind Domänenbegriffe

← [Index](../masterplan.md)

---

Dieses Glossar definiert alle projektspezifischen Begriffe die in der Hivemind-Dokumentation verwendet werden. Alphabetisch sortiert.

---

| Begriff | Definition | Primäre Quelle |
| --- | --- | --- |
| **Code Edge** | Graph-Kante im Nexus Grid. Repräsentiert eine Beziehung zwischen zwei Code Nodes (import, call, dependency, extends). Kann projektübergreifend sein (Monorepo-Support). | [nexus-grid.md](features/nexus-grid.md) |
| **Code Node** | Graph-Knoten im Nexus Grid. Repräsentiert eine Datei, ein Modul, Paket oder Service in der Codebase (`code_nodes`-Tabelle). Hat ein `explored_at`-Timestamp für Fog-of-War-Tracking. | [nexus-grid.md](features/nexus-grid.md) |
| **Command Deck** | Zentrale UI-View mit allen Epics und Tasks inkl. State-Machine-Status, SLA-Monitoring und Human-Action-Buttons (Scoping, Review, Decision Resolution). Ab Phase 2 verfügbar. | [views.md](ui/views.md) |
| **Context Boundary** | Deklarative Scope-Definition pro Task, gesetzt vom Architekt. Beschränkt welche Skills, Docs und externe Services ein Worker sehen darf. Dient der Token-Budget-Optimierung und Prompt-Qualität, nicht der Sicherheit. | [state-machine.md](architecture/state-machine.md) |
| **Context Panel** | Rechte Zone des UI-Layouts. Zeigt standardmäßig die Prompt Station oder die Detail-Ansicht des aktuell gewählten Elements (Task, Skill, etc.). Einklappbar. | [concept.md](ui/concept.md) |
| **Dead Letter Queue (DLQ)** | Auffangbecken für Sync-Einträge die alle Retry-Versuche (default: 5) überschritten haben. Fehlgeschlagene Items wandern von `sync_outbox` in `sync_dead_letter` und erscheinen in der Triage Station zur manuellen Admin-Aktion (Requeue oder Discard). | [sync.md](features/sync.md) |
| **Decision Record** | Nachträgliches Dokumentations-Artefakt, erstellt vom Gaertner nach Task-Abschluss. Hält architektonische Entscheidungen fest (ähnlich ADRs). Kein SLA — dient als Langzeit-Wissen. | [state-machine.md](architecture/state-machine.md) |
| **Decision Request** | Formale Eskalation eines Workers bei Blockade während der Task-Bearbeitung. Setzt den Task atomar auf `blocked` und startet eine SLA-Kette (24h → Owner, 48h → Backup-Owner, 72h → Admin-Eskalation). | [state-machine.md](architecture/state-machine.md) |
| **Discovery Session** | Federation-Koordinationsmechanismus: Ein Kartograph meldet welchen Bereich der Codebase er gerade erkundet (`start_discovery_session`), um Doppelarbeit zwischen Peer-Nodes zu verhindern. Andere Kartographen sehen die aktive Session im Nexus Grid. | [federation.md](features/federation.md) |
| **DoD (Definition of Done)** | Deklarative Checkliste von Akzeptanzkriterien, gesetzt pro Epic (Framework) und verfeinert pro Task. Der Owner prüft DoD-Items manuell im Review-Panel. Unerfüllte DoD führt zu Owner-Reject (`qa_failed`). Gespeichert in `epics.dod_framework` (JSONB). | [state-machine.md](architecture/state-machine.md) |
| **Epic Share** | Federation-Nachrichtentyp (`entity_type = 'epic_share'`). Wird verwendet wenn ein Admin Sub-Tasks eines Epics an einen Peer-Node delegiert. Origin-Node sendet Epic-Spec + zugewiesene Task-Specs an den Peer. | [federation.md](features/federation.md) |
| **Federation** | Betriebsmodus bei dem jeder Entwickler eine souveräne, eigene Hivemind-Instanz betreibt. Nodes tauschen Skills, Wiki-Artikel und delegierte Tasks über signierte HTTP-Nachrichten im VPN aus. Orthogonal zu Solo/Team. | [federation.md](features/federation.md) |
| **Fog of War** | Explorationsmodell des Kartographen. Alle Code-Bereiche starten als unbekannt (Nebel). Nur aktiv analysierte Bereiche werden im Nexus Grid sichtbar. Maximale Zugriffsrechte, aber kein automatisches Wissen. | [kartograph.md](agents/kartograph.md) |
| **Gilde (Guild)** | Die Gesamtheit aller verbundenen Hivemind-Nodes im selben VPN. Jeder Entwickler-Node ist eine unabhängige Basis; die vernetzte Gruppe bildet die Gilde. | [federation.md](features/federation.md) |
| **Guard / Guard-Proposal** | Ausführbare Validierungsregel (Shell-Command, deklarativer Check oder manuelle Bestätigung) die bestanden werden muss bevor ein Task nach `in_review` wechseln kann. Hierarchisch gescopet (global → project → skill → task). Ein Guard-Proposal ist ein zur Überprüfung eingereichter Vorschlag für einen neuen Guard. | [guards.md](features/guards.md) |
| **Hive Station** | Optionaler zentraler Server in der Federation. Bietet Control-Plane-Services (Peer-Discovery, Presence, optional Store-and-Forward-Relay), hat aber **keine** Datenhoheit. Origin-Authority und Ed25519-Signaturen bleiben bei den Nodes. | [overview.md](architecture/overview.md) |
| **Inline-Review / Inline-Scoping** | Vereinfachte UI-Formulare direkt in der Prompt Station während Phase 1 (vor Existenz des Command Decks). Inline-Review: Mini-DoD-Checkliste mit Approve/Reject. Inline-Scoping: Minimalformular (Priorität, Deadline) für `incoming`-Epics. Ab Phase 2 Delegation an Command Deck. | [phase-1.md](phases/phase-1.md) |
| **Key-Rotation / Revocation** | Federation-Sicherheitsprozeduren. **Rotation:** Neues Ed25519-Keypair, Public-Key-Verteilung an Peers, Grace-Period für alten Key (default: 1h). **Revocation:** Sofortige Invalidierung eines Peer Public Keys (Key → NULL), alle weiteren Nachrichten werden abgelehnt. | [federation.md](features/federation.md) |
| **Mercenary** | Spielmetapher-Name für einen Worker-Agent (AI-Client). Im Sci-Fi-HUD erhält jeder Worker als "Söldner" ein Mission-Briefing (Prompt) und ein Loadout (Skills) vor der Quest (Task). | [federation.md](features/federation.md) |
| **Mercenary Loadout** | Das spezifische Skill-Set (gepinnt auf exakte Versionen), das vor einer Quest an einen Worker angeheftet wird. Der Architekt pinnt Skills; die Prompt Station zeigt das resultierende Loadout mit Token-Counts und Guards. | [federation.md](features/federation.md) |
| **Nexus Grid** | Interaktive 2D/3D-Graph-Visualisierung der Codebase. Kombiniert Code-Struktur (Nodes/Edges), Kartograph-Explorationsstatus (Fog-of-War-Overlay) und Bug-Report-Dichte (Heatmap). Unterstützt Multi-Projekt-Views. | [nexus-grid.md](features/nexus-grid.md) |
| **Origin-Authority** | Souveränitätsprinzip in der Federation: Jede Entität (Epic, Skill, Wiki-Artikel) hat eine `origin_node_id`. Nur der Ursprungs-Node darf die Entität editieren. Peers erhalten Read-only-Kopien. | [overview.md](architecture/overview.md) |
| **Outbox Pattern** | Asynchroner Nachrichtenzustellungsmechanismus für alle externen Sync-Operationen. Jeder ausgehende Write (YouTrack, Sentry, Federation) wird erst in `sync_outbox` persistiert, dann mit Exponential-Backoff-Retry verarbeitet. Garantiert At-least-once-Delivery. | [sync.md](features/sync.md) |
| **Peer / Peer Node** | Eine Remote-Hivemind-Instanz in der Federation. Jeder Peer ist souverän (eigene DB, eigenes Docker Compose), identifiziert via UUID, Name, VPN-URL und Ed25519 Public Key. Kommunikation über signierte HTTP/REST-Endpoints. | [federation.md](features/federation.md) |
| **Prompt Station** | Kernstück der BYOAI-Interaktion (UI-View). Erkennt welcher Agent als nächstes im Workflow benötigt wird, zeigt den fertigen Prompt, und lässt den User ihn in seinen AI-Client kopieren. In Phase 8 wird sie zur automatisierten Monitoring-View. | [prompt-station.md](ui/prompt-station.md) |
| **Skill / Skill Composition** | Versioniertes, agent-agnostisches Instruction-Dokument (Markdown + YAML-Frontmatter). Beschreibt *wie* ein bestimmter Task-Typ angegangen wird. **Skill Composition** ("Stacking") erlaubt Vererbung via `extends` (max 3 Ebenen, max 5 Parents). Content wird angehängt, Guards werden geerbt. | [skills.md](features/skills.md) |
| **SLA (Service Level Agreement)** | Zeitbezogene Constraints auf Epics und Decision Requests. Epic-SLA (`sla_due_at`) steuert UI-Dringlichkeit und Prompt-Queue-Priorisierung. Decision-Request-SLA folgt einer 24h/48h/72h-Eskalationskette. | [state-machine.md](architecture/state-machine.md) |
| **Solo / Team / Federation** | Die drei Betriebsmodi. **Solo:** Einzelnutzer, RBAC deaktiviert, Self-Review erzwungen. **Team:** Geteilte zentrale Instanz mit vollem RBAC. **Federation:** Jeder Entwickler betreibt eigenen souveränen Node. Solo/Team per `hivemind_mode` umschaltbar; Federation orthogonal via `HIVEMIND_FEDERATION_ENABLED`. | [overview.md](architecture/overview.md) |
| **Triage Station** | Admin-only UI-View als Eingangstor für alle Events die nicht auto-geroutet werden können: ungeroutete externe Events, Skill/Guard-Proposals, Epic-Restructure-Proposals, Dead Letters, eskalierte Tasks. Priorisiert nach Dringlichkeit. | [triage.md](agents/triage.md) |
| **Trust Boundary** | Sicherheitsperimeter zwischen AI-Client/Chat und Hivemind-Backend. Alle MCP-Calls von externen AI-Clients werden als potenziell unsicher behandelt. Nur die Hivemind-Middleware darf validieren, autorisieren und Writes committen. | [overview.md](architecture/overview.md) |
| **Wizard of Oz (Pattern)** | BYOAI-Designphilosophie in Phase 1–7: Das System generiert strukturierte Prompts, aber ein Mensch kopiert sie manuell in einen AI-Client. Aus Systemperspektive ist der Workflow identisch zu autonomem Betrieb — selbe Endpoints, selbe Datenverträge — aber der Mensch ist der Intermediär. | [bibliothekar.md](agents/bibliothekar.md) |
