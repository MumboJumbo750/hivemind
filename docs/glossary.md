# Glossar — Hivemind Domänenbegriffe

← [Index](../masterplan.md)

---

Dieses Glossar definiert alle projektspezifischen Begriffe die in der Hivemind-Dokumentation verwendet werden. Alphabetisch sortiert.

---

| Begriff | Definition | Primäre Quelle |
| --- | --- | --- |
| **Architekt (Agent)** | Agent/Workflow-Funktion die Epics in konkrete Tasks zerlegt. Liest Epic-Spec, DoD-Framework und Skill-Pool; erstellt Task-Breakdown mit Context Boundaries und pinned Skills. Keine eigene RBAC-Rolle — nutzt `developer`- oder `admin`-Rechte. Gegenstück zum Strategen (top-down) und Kartographen (bottom-up). | [state-machine.md](architecture/state-machine.md) |
| **Bibliothekar (Agent)** | Agent/Workflow-Funktion für Context Assembly. Sammelt alle relevanten Kontextblöcke (Skill-Content, Docs, Epic-Summary, Guards) und assembliert daraus den fertigen Prompt für den jeweiligen Agent. In Phase 1–2 als statischer Prompt-Generator ohne Backend-Service; ab Phase 3 als vollständiger Prompt-Assembly-Service. | [bibliothekar.md](agents/bibliothekar.md) |
| **Code Edge** | Graph-Kante im Nexus Grid. Repräsentiert eine Beziehung zwischen zwei Code Nodes (import, call, dependency, extends). Kann projektübergreifend sein (Monorepo-Support). | [nexus-grid.md](features/nexus-grid.md) |
| **Code Node** | Graph-Knoten im Nexus Grid. Repräsentiert eine Datei, ein Modul, Paket oder Service in der Codebase (`code_nodes`-Tabelle). Hat ein `explored_at`-Timestamp für Fog-of-War-Tracking. | [nexus-grid.md](features/nexus-grid.md) |
| **Command Deck** | Zentrale UI-View mit allen Epics und Tasks inkl. State-Machine-Status, SLA-Monitoring und Human-Action-Buttons (Scoping, Review, Decision Resolution). Ab Phase 2 verfügbar. | [views.md](ui/views.md) |
| **Context Boundary** | Deklarative Scope-Definition pro Task, gesetzt vom Architekt. Beschränkt welche Skills, Docs und externe Services ein Worker sehen darf. Dient der Token-Budget-Optimierung und Prompt-Qualität, nicht der Sicherheit. | [state-machine.md](architecture/state-machine.md) |
| **Conductor** | Autonomes Orchestrierungssystem in Phase 8. Horcht auf System-Events (Task-State-Transitions, Epic-Status etc.) und dispatcht automatisch den passenden Agent (via `conductor_dispatches`-Tabelle als Audit-Trail). Governance-Level steuert ob der Conductor vollautomatisch handelt (`auto`), eine Empfehlung gibt (`assisted`) oder nur vorschlägt (`manual`). | [phase-8.md](phases/phase-8.md) |
| **Context Panel** | Rechte Zone des UI-Layouts. Zeigt standardmäßig die Prompt Station oder die Detail-Ansicht des aktuell gewählten Elements (Task, Skill, etc.). Einklappbar. | [concept.md](ui/concept.md) |
| **Dead Letter Queue (DLQ)** | Auffangbecken für Sync-Einträge die alle Retry-Versuche (default: 5) überschritten haben. Fehlgeschlagene Items wandern von `sync_outbox` in `sync_dead_letter` und erscheinen in der Triage Station zur manuellen Admin-Aktion (Requeue oder Discard). | [sync.md](features/sync.md) |
| **Decision Record** | Nachträgliches Dokumentations-Artefakt, erstellt vom Gaertner nach Task-Abschluss. Hält architektonische Entscheidungen fest (ähnlich ADRs). Kein SLA — dient als Langzeit-Wissen. | [state-machine.md](architecture/state-machine.md) |
| **Decision Request** | Formale Eskalation eines Workers bei Blockade während der Task-Bearbeitung. Setzt den Task atomar auf `blocked` und startet eine SLA-Kette (24h → Owner, 48h → Backup-Owner, 72h → Admin-Eskalation). | [state-machine.md](architecture/state-machine.md) |
| **Discovery Session** | Federation-Koordinationsmechanismus: Ein Kartograph meldet welchen Bereich der Codebase er gerade erkundet (`hivemind/start_discovery_session`), um Doppelarbeit zwischen Peer-Nodes zu verhindern. Andere Kartographen sehen die aktive Session im Nexus Grid. | [federation.md](features/federation.md) |
| **DoD (Definition of Done)** | Deklarative Checkliste von Akzeptanzkriterien, gesetzt pro Epic (Framework) und verfeinert pro Task. Der Owner prüft DoD-Items manuell im Review-Panel. Unerfüllte DoD führt zu Owner-Reject (`qa_failed`). Gespeichert in `epics.dod_framework` (JSONB). | [state-machine.md](architecture/state-machine.md) |
| **Epic Share** | Federation-Nachrichtentyp (`entity_type = 'epic_share'`). Wird verwendet wenn ein Admin Sub-Tasks eines Epics an einen Peer-Node delegiert. Origin-Node sendet Epic-Spec + zugewiesene Task-Specs an den Peer. | [federation.md](features/federation.md) |
| **Epic Proposal** | Strategiebasierter Vorschlag für ein neues Epic, erstellt vom Strategen via `propose_epic`. Enthält Titel, Beschreibung, Rationale, vorgeschlagene Priorität, Phase und Abhängigkeiten. Muss von Admin/Owner in der Triage Station akzeptiert werden bevor ein Epic entsteht (`state: proposed → accepted`). | [stratege.md](agents/stratege.md) |
| **Federation** | Betriebsmodus bei dem jeder Entwickler eine souveräne, eigene Hivemind-Instanz betreibt. Nodes tauschen Skills, Wiki-Artikel und delegierte Tasks über signierte HTTP-Nachrichten im VPN aus. Orthogonal zu Solo/Team. | [federation.md](features/federation.md) |
| **Fog of War** | Explorationsmodell des Kartographen. Alle Code-Bereiche starten als unbekannt (Nebel). Nur aktiv analysierte Bereiche werden im Nexus Grid sichtbar. Maximale Zugriffsrechte, aber kein automatisches Wissen. | [kartograph.md](agents/kartograph.md) |
| **Gaertner (Agent)** | Agent/Workflow-Funktion für Wissenskonsolidierung nach Task-Abschluss. Analysiert das Task-Ergebnis und destilliert daraus wiederverwendbares Wissen: neue Skills (via `propose_skill`), Skill-Verbesserungen (via `propose_skill_change`), Wiki-Artikel (via `create_wiki_article`) und Decision Records. Aktiv ab Phase 5. | [phase-5.md](phases/phase-5.md) |
| **Governance-Level** | Dreistufige Automatisierungseinstellung pro Entscheidungstyp in Phase 8. `manual` = immer menschliche Entscheidung, `assisted` = AI-Empfehlung + menschliche Bestätigung, `auto` = vollautomatisch mit optionaler Grace-Period für Veto. Konfigurierbar per Entscheidungstyp (review, epic_proposal, skill_merge etc.) via `PATCH /api/settings/governance`. | [rest-api.md](architecture/rest-api.md) |
| **Gilde (Guild)** | Die Gesamtheit aller verbundenen Hivemind-Nodes im selben VPN. Jeder Entwickler-Node ist eine unabhängige Basis; die vernetzte Gruppe bildet die Gilde. | [federation.md](features/federation.md) |
| **Guard / Guard-Proposal** | Ausführbare Validierungsregel (Shell-Command, deklarativer Check oder manuelle Bestätigung) die bestanden werden muss bevor ein Task nach `in_review` wechseln kann. Hierarchisch gescopet (global → project → skill → task). Ein Guard-Proposal ist ein zur Überprüfung eingereichter Vorschlag für einen neuen Guard. | [guards.md](features/guards.md) |
| **Hive Station** | Optionaler zentraler Server in der Federation. Bietet Control-Plane-Services (Peer-Discovery, Presence, optional Store-and-Forward-Relay), hat aber **keine** Datenhoheit. Origin-Authority und Ed25519-Signaturen bleiben bei den Nodes. | [overview.md](architecture/overview.md) |
| **Inline-Review / Inline-Scoping** | Vereinfachte UI-Formulare direkt in der Prompt Station während Phase 1 (vor Existenz des Command Decks). Inline-Review: Mini-DoD-Checkliste mit Approve/Reject. Inline-Scoping: Minimalformular (Priorität, Deadline) für `incoming`-Epics. Ab Phase 2 Delegation an Command Deck. | [phase-1.md](phases/phase-1.md) |
| **Kartograph (Agent)** | Agent/Workflow-Funktion für Code-Exploration. Analysiert Codebase-Bereiche, erstellt Code-Nodes im Nexus Grid (Fog-of-War aufdecken), generiert Wiki-Artikel und verknüpft Code-Strukturen mit Epics. Nutzt die RBAC-Rolle `kartograph` (erweiterte Lese- und Write-Rechte für Code-Nodes und Wiki). In Federation: teilt Discoveries mit Peers via `discovery_session` um Doppelarbeit zu vermeiden. | [kartograph.md](agents/kartograph.md) |
| **Key-Rotation / Revocation** | Federation-Sicherheitsprozeduren. **Rotation:** Neues Ed25519-Keypair, Public-Key-Verteilung an Peers, Grace-Period für alten Key (default: 1h). **Revocation:** Sofortige Invalidierung eines Peer Public Keys (Key → NULL), alle weiteren Nachrichten werden abgelehnt. | [federation.md](features/federation.md) |
| **Memory Ledger** | Persistentes Wissensregister das alle destillierten Erkenntnisse speichert. Besteht aus Skills (prozedurales Wissen), Wiki-Artikeln (konzeptuelles Wissen) und Decision Records (architektonische Entscheidungen). Wird durch den Gaertner-Flow befüllt und durch den Bibliothekar beim Prompt-Assembly gelesen. Ermöglicht Zero-Loss-Memory über Task-Grenzen hinweg. | [features/skills.md](features/skills.md) |
| **Mercenary** | Spielmetapher-Name für einen Worker-Agent (AI-Client). Im Sci-Fi-HUD erhält jeder Worker als "Söldner" ein Mission-Briefing (Prompt) und ein Loadout (Skills) vor der Quest (Task). | [federation.md](features/federation.md) |
| **Mercenary Loadout** | Das spezifische Skill-Set (gepinnt auf exakte Versionen), das vor einer Quest an einen Worker angeheftet wird. Der Architekt pinnt Skills; die Prompt Station zeigt das resultierende Loadout mit Token-Counts und Guards. | [federation.md](features/federation.md) |
| **Nexus Grid** | Interaktive 2D/3D-Graph-Visualisierung der Codebase. Kombiniert Code-Struktur (Nodes/Edges), Kartograph-Explorationsstatus (Fog-of-War-Overlay) und Bug-Report-Dichte (Heatmap). Unterstützt Multi-Projekt-Views. | [nexus-grid.md](features/nexus-grid.md) |
| **Origin-Authority** | Souveränitätsprinzip in der Federation: Jede Entität (Epic, Skill, Wiki-Artikel) hat eine `origin_node_id`. Nur der Ursprungs-Node darf die Entität editieren. Peers erhalten Read-only-Kopien. | [overview.md](architecture/overview.md) |
| **Outbox Pattern** | Asynchroner Nachrichtenzustellungsmechanismus für alle externen Sync-Operationen. Jeder ausgehende Write (YouTrack, Sentry, Federation) wird erst in `sync_outbox` persistiert, dann mit Exponential-Backoff-Retry verarbeitet. Garantiert At-least-once-Delivery. | [sync.md](features/sync.md) |
| **Peer / Peer Node** | Eine Remote-Hivemind-Instanz in der Federation. Jeder Peer ist souverän (eigene DB, eigenes Docker Compose), identifiziert via UUID, Name, VPN-URL und Ed25519 Public Key. Kommunikation über signierte HTTP/REST-Endpoints. | [federation.md](features/federation.md) |
| **Prompt Station** | Kernstück der BYOAI-Interaktion (UI-View). Erkennt welcher Agent als nächstes im Workflow benötigt wird, zeigt den fertigen Prompt, und lässt den User ihn in seinen AI-Client kopieren. In Phase 8 wird sie zur automatisierten Monitoring-View. | [prompt-station.md](ui/prompt-station.md) |
| **Reviewer (Agent)** | Agent/Workflow-Funktion in Phase 8 (Autonomy). Analysiert ein Task-Ergebnis automatisch gegen DoD-Checkliste und Guards; gibt eine Empfehlung (`approve` / `reject` / `needs_human_review`) mit Confidence-Score zurück. Gespeichert in `review_recommendations`. Im `assisted` Governance-Level bestätigt der Owner die Empfehlung; im `auto`-Level handelt der Conductor direkt (mit optionaler Grace-Period für Veto). | [rest-api.md](architecture/rest-api.md) |
| **Skill / Skill Composition** | Versioniertes, agent-agnostisches Instruction-Dokument (Markdown + YAML-Frontmatter). Beschreibt *wie* ein bestimmter Task-Typ angegangen wird. **Skill Composition** ("Stacking") erlaubt Vererbung via `extends` (max 3 Ebenen, max 5 Parents). Content wird angehängt, Guards werden geerbt. | [skills.md](features/skills.md) |
| **SLA (Service Level Agreement)** | Zeitbezogene Constraints auf Epics und Decision Requests. Epic-SLA (`sla_due_at`) steuert UI-Dringlichkeit und Prompt-Queue-Priorisierung. Decision-Request-SLA folgt einer 24h/48h/72h-Eskalationskette. | [state-machine.md](architecture/state-machine.md) |
| **Stratege** | Agent/Workflow-Funktion für strategische Planung. Analysiert Planungsdokumente (Masterpläne, PRDs, RFCs) und leitet daraus Epic-Proposals mit Abhängigkeiten und Phasen-Zuordnung ab. Arbeitet top-down (Plan → Epics), im Gegensatz zum Kartograph (bottom-up: Code → Verständnis) und Architekt (middle-out: Epic → Tasks). Keine eigene RBAC-Rolle — nutzt `developer`- oder `admin`-Rechte. | [stratege.md](agents/stratege.md) |
| **Solo / Team / Federation** | Die drei Betriebsmodi. **Solo:** Einzelnutzer, RBAC deaktiviert, Self-Review erzwungen. **Team:** Geteilte zentrale Instanz mit vollem RBAC. **Federation:** Jeder Entwickler betreibt eigenen souveränen Node. Solo/Team per `hivemind_mode` umschaltbar; Federation orthogonal via `HIVEMIND_FEDERATION_ENABLED`. | [overview.md](architecture/overview.md) |
| **Token Budget** | Maximale Anzahl von Tokens die ein Prompt verbrauchen darf. Konfigurierbar als Global-Default (`token_budget_default`, Standard: 8000), per Agent-Rolle (`ai_provider_configs.token_budget`) und per Task-Context-Boundary (`max_token_budget`). Der Bibliothekar priorisiert Context-Blöcke so dass das Budget nicht überschritten wird: Skills > Docs (neueste zuerst). Der Token Radar in der UI zeigt den aktuellen Budget-Verbrauch an. | [phases/overview.md](phases/overview.md) |
| **Triage Station** | Admin-only UI-View als Eingangstor für alle Events die nicht auto-geroutet werden können: ungeroutete externe Events, Skill/Guard-Proposals, Epic-Restructure-Proposals, Dead Letters, eskalierte Tasks. Priorisiert nach Dringlichkeit. | [triage.md](agents/triage.md) |
| **Trust Boundary** | Sicherheitsperimeter zwischen AI-Client/Chat und Hivemind-Backend. Alle MCP-Calls von externen AI-Clients werden als potenziell unsicher behandelt. Nur die Hivemind-Middleware darf validieren, autorisieren und Writes committen. | [overview.md](architecture/overview.md) |
| **Worker (Agent)** | Agent/Workflow-Funktion die eine konkrete Quest (Task) technisch umsetzt. Erhält ein Mission-Briefing (Prompt) mit gepinnten Skills (Loadout), Context Boundaries und Guards. Liefert `result` + `artifacts` via `submit_result` und meldet Fertigstellung via `update_task_state → in_review`. Im Sprachbild: Mercenary der vom Kommandanten ausgerüstet wird. RBAC-Rolle: `developer`. | [state-machine.md](architecture/state-machine.md) |
| **Wizard of Oz (Pattern)** | BYOAI-Designphilosophie in Phase 1–7: Das System generiert strukturierte Prompts, aber ein Mensch kopiert sie manuell in einen AI-Client. Aus Systemperspektive ist der Workflow identisch zu autonomem Betrieb — selbe Endpoints, selbe Datenverträge — aber der Mensch ist der Intermediär. | [bibliothekar.md](agents/bibliothekar.md) |
| **Circuit Breaker** | Fehlertoleranz-Pattern im Embedding-Service. Nach N aufeinanderfolgenden Ollama-Timeouts wechselt der Breaker in den `OPEN`-State (neue Embedding-Requests sofort mit `NULL` beantwortet, Feature-Degradation statt Fehler). Nach adaptivem exponentiellen Backoff (60s → 120s → ... → max 600s) wird ein Probe-Request gesendet. Bei Erfolg: `CLOSED`, Queue-Verarbeitung fortgesetzt. | [phase-3.md](phases/phase-3.md) |
| **Epic Restructure Proposal** | Vorschlag des Kartographen zur Restrukturierung der Epic-Hierarchie wenn Code-Organisation und Epic-Struktur nicht übereinstimmen. Typen: `split` (ein Epic → mehrere), `merge` (mehrere → eines), `task_move` (Tasks zwischen Epics verschieben). Lifecycle: `proposed → accepted → applied` oder `rejected`. Nur Tasks in verschiebbaren States (`incoming`, `scoped`, `ready`, `blocked`) können Teil eines Restructures sein. | [epic-restructure.md](features/epic-restructure.md) |
| **Gamification / EXP** | Punkte- und Belohnungssystem das echte Entwicklerarbeit in Spielmechaniken übersetzt. EXP wird für Tasks, Reviews, Skills, Code-Exploration und Federation-Aktivitäten vergeben. Level 1–10 mit linearen Schwellwerten. Badges für besondere Errungenschaften. EXP-Logik aktiv ab Phase 5; Seed-Daten (Tabellen, Level-Definitionen) ab Phase 1. | [gamification.md](features/gamification.md) |
| **Guard-Reset** | Mechanismus der bei `qa_failed → in_progress`-Transition ausgelöst wird: Alle `task_guards`-Einträge für den Task werden auf `status = 'pending'` zurückgesetzt — unabhängig vom Vorher-Status (`pending`, `failed`, `skipped`). Manuell geskippte Guards müssen erneut vom Owner bestätigt werden. | [state-machine.md](architecture/state-machine.md) |
| **Skill Deprecation** | Letzte Lebensphase eines Skills. Deprecated Skills werden dem Bibliothekar nicht mehr zur Auswahl angeboten, bleiben aber lesbar für Audit und Historien-Zwecke. Tasks die auf eine deprecated Skill-Version gepinnt sind, werden nicht automatisch benachrichtigt — die Version bleibt gültig bis Task-Abschluss. Neue Task-Verknüpfungen auf deprecated Skills sind nicht möglich. | [state-machine.md](architecture/state-machine.md) |
| **Observability** | Logging-, Metrics- und Tracing-Stack für Hivemind. Pflicht: strukturiertes JSON-Logging (`INFO`-Level in Produktion). Optional: Prometheus-Metrics-Endpoint (`/metrics`) und OpenTelemetry-Tracing. Health-Check via `GET /health`. Slow-Query-Warning bei DB-Queries > 1000ms. | [observability.md](architecture/observability.md) |
| **Disaster Recovery** | Backup- und Wiederherstellungs-Strategie. Vollbackup täglich via `pg_dump` (7 tägliche + 4 wöchentliche + 3 monatliche). Optional: PITR via WAL-Archivierung (empfohlen für Produktion). Separate Backup-Volumes zwingend. Federation-Key-Export vor jedem Deployment. RTO: 30–120 Min; RPO: < 24h (ohne PITR) / < 5 Min (mit PITR). | [disaster-recovery.md](architecture/disaster-recovery.md) |
| **Stream Token** | Kurzlebiger (30s), single-use Authentifizierungstoken speziell für SSE-Verbindungen. Da SSE keine Custom-Header unterstützt, kann kein reguläres JWT als Query-Parameter gesendet werden (sichtbar in Logs). Der Stream-Token wird via `POST /api/auth/stream-token` generiert und als `?token=` Query-Parameter beim SSE-Handshake übergeben. Nach einmaliger Verwendung invalidiert. | [rest-api.md](architecture/rest-api.md) |
| **Sync-Strike / Co-op Bonus** | Gamification-Mechanik: Wenn zwei verbundene Nodes gleichzeitig an Tasks desselben Shared Epics arbeiten, erhalten beide für die Session-Dauer leuchtende Avatar-Grenzen im UI (visueller Effekt). Kein Gameplay-Vorteil — rein kosmetisches Koop-Signal. Implementiert als SSE-Event `peer_sync_strike` das ausgelöst wird wenn zwei `task_state_changed → in_progress`-Events für dasselbe Epic innerhalb von 5 Minuten auf unterschiedlichen Nodes einlaufen. | [concept.md](ui/concept.md) |
